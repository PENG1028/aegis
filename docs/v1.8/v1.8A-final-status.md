## 4. Risk Code Final Definition

### 4.1 Risk Code Table

| Code | Severity | Trigger | Detected By |
|------|----------|---------|-------------|
| `PUBLIC_DOMAIN_BOUNCE` | warning | DNS resolves to public IP, no Aegis route | `TraceEgress` |
| `PUBLIC_TARGET_EGRESS` | warning | Route targets a public IP | `CheckRouteSafety`, `GetPlannerWarnings` |
| `GATEWAY_LINK_BYPASS_RISK` | warning | Public target without GatewayLink | `CheckRouteSafety`, `GetPlannerWarnings` |
| `SELF_LOOP` | error | Target points to a gateway listener (host + port) | `CheckRouteSafety`, `GetPlannerWarnings` |
| `DOMAIN_RESOLVES_TO_SELF` | error | DNS resolves to gateway's own IP | `TraceEgress` |
| `INTERNAL_TARGET_AVAILABLE` | info | Private/loopback target (suggested) | Reserved |
| `UNKNOWN_DOMAIN` | info | No route, no managed domain, no DNS | `TraceEgress` |

### 4.2 SELF_LOOP Final Semantics

`SELF_LOOP` fires **only** when ALL conditions are met:

1. **Host** is a loopback address OR matches a current node IP
2. **Port** is a registered gateway listener port:
   - From `listeners` table (active/planned status)
   - Fallback: `[80, 443, 8443]`

**Safe loopback targets (no SELF_LOOP):**
- `127.0.0.1:3001` — loopback, not a listener port → classification: `loopback`, safe
- `localhost:8080` — loopback, not a listener port → safe

**Self-loop targets (SELF_LOOP):**
- `127.0.0.1:80` — loopback, listener port → `SELF_LOOP` error
- `node_public:443` — node IP, listener port → `SELF_LOOP` error
- `node_private:8443` — node IP, listener port → `SELF_LOOP` error

### 4.3 Semantic Non-Overlap Guarantee

| Scenario | Risk Codes | Room for Confusion |
|----------|------------|-------------------|
| Unknown domain, no DNS | `UNKNOWN_DOMAIN` only | None — distinct trigger |
| DNS public, no route | `PUBLIC_DOMAIN_BOUNCE` only | None |
| Route public, no GWLink | `PUBLIC_TARGET_EGRESS` + `GATEWAY_LINK_BYPASS_RISK` | Co-occurrence is by design |
| Route public, GWLink | `PUBLIC_TARGET_EGRESS` only (by design) | None |
| DNS resolves to self | `DOMAIN_RESOLVES_TO_SELF` only | None |
| Route targets gateway listener | `SELF_LOOP` only | None |
| Non-listener node target | No risks | None |

## 6. Planner Warning Behavior

Safety warnings from `GetPlannerWarnings()` are integrated into the apply `Plan()` cycle:

```go
// In resolveRouteConfigWithService (per route):
safetyRisks := p.safetySvc.GetPlannerWarnings(domain, targetHost, gatewayLinkID)
for _, risk := range safetyRisks {
    warnings = append(warnings, ApplyWarning{
        Code: "SAFETY_" + risk.Code, Severity: risk.Severity, ...
    })
}
```

**Key behavior:**
- Warnings are generated during `Plan()` → stored in `ApplyPlan.Warnings`
- Warnings NEVER block apply — `Apply()` continues past `Plan()` unconditionally
- Warning codes are prefixed with `SAFETY_` (e.g., `SAFETY_GATEWAY_LINK_BYPASS_RISK`)
- See `docs/v1.8/planner-warning-real-smoke.md` for proof

## 7. Not Supported (v1.8A Scope)

v1.8A is **detection only**. The following are explicitly out of scope:

| Item | Reason |
|------|--------|
| ❌ Enforcement / blocking unsafe routes | v1.8A is detection-only |
| ❌ iptables / nftables manipulation | OS-level, out of scope |
| ❌ Transparent proxy / DNS hijack | Would violate detection-only principle |
| ❌ eBPF / sidecar mesh / service mesh | Too invasive, deferred |
| ❌ Background monitoring loop | Would need state management, out of scope |
| ❌ Alert system | Would need notification channels |
| ❌ UI | No UI in any v1.x phase |
| ❌ Multi-admin / SaaS | Personal gateway only |
| ❌ Billing / quota engine | Never planned |
