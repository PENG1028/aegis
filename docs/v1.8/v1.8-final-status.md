## 4. Final Relay Semantics

These are the definitive semantics of the Managed Egress Relay HTTP system as of v1.8B:

### Dispatch Rules

1. **Aegis managed HTTP targets are never directly connected** to the remote target port. All traffic goes through the gateway proxy.

2. **Callers always access the target node's gateway listener** (port 80 or 443). Direct host:port access is blocked by the cloud security group.

3. **The target node's gateway proxies to Aegis relay handler** (currently via Caddy → `127.0.0.1:9000/__aegis/relay`).

4. **The relay handler dispatches to the local target** only: `127.0.0.1:<target_port>`. No remote host forwarding.

### Auth & Routing

5. **endpoint.node_id must equal the current node.** A request for a route whose endpoint belongs to a different node is rejected with 409 CONFLICT. Empty node_id is also rejected (must be explicitly set).

6. **Cross-node relay requires GatewayLink authentication.** The `X-Aegis-Gateway-ID` and `X-Aegis-Gateway-Token` headers are validated by the receiving node. Token is encrypted-at-rest with AES-256-GCM.

### Security Constraints

7. **`unavailable` mode does NOT fallback to `remote_target_host:target_port`.** Direct target suppression is always active. The caller receives `MODE_UNAVAILABLE` with `direct_target_suppressed = true`.

8. **`X-Aegis-Target-Host` and `X-Aegis-Target-Port` headers are rejected** with 400 BAD REQUEST. This prevents open proxy abuse.

9. **Hop count is limited to 1.** Any request with `X-Aegis-Hop > 1` is rejected with 508 LOOP DETECTED.

10. **The relay is NOT an open proxy.** Only authenticated, route-bound requests to local endpoints are forwarded.

### Error Codes

| Error | HTTP Status | When |
|-------|-------------|------|
| `MISSING_ROUTE_ID` | 400 | Missing X-Aegis-Route-ID |
| `MISSING_GATEWAY_ID` | 400 | Missing X-Aegis-Gateway-ID |
| `MISSING_GATEWAY_TOKEN` | 400 | Missing X-Aegis-Gateway-Token |
| `MISSING_SOURCE_NODE` | 400 | Missing X-Aegis-Source-Node |
| `INVALID_GATEWAY` | 401 | Gateway ID not found |
| `INVALID_GATEWAY_TOKEN` | 403 | Token verification failed |
| `UNKNOWN_SOURCE_NODE` | 403 | Source node not recognized |
| `ROUTE_NOT_FOUND` | 404 | Route ID not in database |
| `NO_ENDPOINTS` | 404 | Service has no enabled endpoints |
| `ENDPOINT_NODE_UNKNOWN` | 409 | Endpoint has empty node_id |
| `ENDPOINT_NOT_LOCAL` | 409 | Endpoint belongs to another node |
| `TARGET_HEADER_REJECTED` | 400 | X-Aegis-Target-Host/Port present |
| `MAX_HOPS_EXCEEDED` | 508 | Hop > 1 |
| `TARGET_UNREACHABLE` | 502 | Local target host:port unreachable |

## 5. Final Secret Semantics

These are the definitive semantics of GatewayLink secret storage as of v1.8B-5/6:

### Storage

1. **SQLite no longer stores GatewayLink raw tokens in plaintext.** The `encrypted_secret` column contains AES-256-GCM ciphertext.

2. **AES-256-GCM authenticated encryption** is used with random 12-byte nonces. Tampered ciphertext or wrong key causes GCM authentication failure.

3. **Three storage columns**: `encrypted_secret` (base64 ciphertext), `secret_nonce` (base64 nonce), `secret_version` (integer, incremented on rotation).

### Master Key

4. **Master key sources** (checked in order):
   - `AEGIS_SECRET_KEY` environment variable (64 hex chars)
   - `/etc/aegis/secret.key` file (0600 or 0640 permissions)
   - Dev mode: auto-generated ephemeral key

5. **Key validation**: exactly 32 bytes (64 hex chars). Invalid key → clear error. Missing key in production → service starts with warning, legacy HMAC fallback only.

6. **Key lifecycle**: read once at startup, held in memory for process lifetime. Key rotation (changing the master key itself) is deferred.

### Token Exposure

7. **Raw token is returned only once** — in the HTTP response of `POST /api/admin/v1/gateway-links` (create) and `POST /api/admin/v1/gateway-links/{id}/rotate`.

8. **List/Get/Trace/Log never expose the raw token.** All secret fields have `json:"-"` tags. `Service.List()` clears secret fields entirely. `Service.Get()` returns version metadata but never the raw value.

9. **Planner/render decrypts in memory only** for Caddy config generation. The decrypted token is never written to disk by Aegis itself.

### Known Boundary

10. **The rendered Caddyfile config still contains the raw token** in `header_up X-Aegis-Gateway-Token` directives. This is a known boundary: the token exists on disk in the proxy's config file, not in Aegis's database.

### Key Loss

11. **If the master key is lost, encrypted secrets become undecryptable.** The HMAC fallback in `auth_value` still allows legacy auth comparison (if the token is known to the caller), but `GetRawSecret()` will return an error for encrypted gateways.

12. **Backup SQLite alongside the master key.** Loss of either means loss of encrypted data.

### Grace Period (Rotation)

13. **No grace period on rotation.** After `RotateSecretEncrypted()` completes, the old token is immediately invalidated on the issuing node. The receiving node must be updated independently (manual two-node sync required).

14. **HMAC fallback during rotation**: on the receiving node, old token may still be accepted if the HMAC hash has not been updated. This is not a grace period — it's a consequence of independent DBs.

## 7. Known Boundaries

### HTTP Relay Only

The managed egress relay currently supports **HTTP only**. Raw TCP, CONNECT tunnel, and WebSocket tunnel are explicitly out of scope for v1.8B. The relay handler uses `net/http` reverse proxy semantics.

### Raw Token in Caddyfile

The rendered Caddyfile config still contains the raw GatewayLink token in `header_up X-Aegis-Gateway-Token` directives. This is a **known boundary**: the token exists in plaintext in the gateway proxy's config file, on disk, not in Aegis's database.

Mitigation: the Caddyfile is on the same server as Aegis and has restricted permissions. Future work could use Caddy's `{env.*}` placeholders or an external secrets manager.

### Legacy HMAC Fallback

The legacy HMAC hash in `auth_value` is a **deterministic hash** of the raw token with a hardcoded HMAC key (`"aegis-gateway-link-v1"`). This is a degraded compatibility mode:

- New gateways use AES-256-GCM (no HMAC dependency)
- Legacy gateways work via HMAC fallback
- If the hardcoded HMAC key were compromised, existing legacy HMAC hashes could be brute-forced
- Legacy HMAC should be phased out in a future version

### No Multi-node Control Plane

Each node has its own independent SQLite database. GatewayLink secrets are not synchronized between nodes. Rotation requires manual two-node sync (update both Server A and Server B independently).

### No UI

No web UI in v1.8B. All operations are via CLI (`aegis`) or REST API.

### No Grace Period on Token Rotation

After rotation, the old token is immediately invalidated on the issuing node. The receiving node must be updated independently. There is no automatic cooldown or dual-token window.

## 10. Operational Notes

### Startup Warnings

When the master key is not available, Aegis prints:
```
warning: master key not available — gateway link secrets will use legacy HMAC storage
```
This is expected for CLI commands running as non-root. The server process (running as root) loads the key from `/etc/aegis/secret.key`.

### Master Key File Requirements

- **Path**: `/etc/aegis/secret.key`
- **Permissions**: `0600` or `0640`
- **Owner**: `root:root`
- **Content**: exactly 64 hex characters (32 bytes)

### Backup Strategy

The master key must be backed up alongside the SQLite database:
- SQLite: `/home/ubuntu/.aegis/aegis.db`
- Key: `/etc/aegis/secret.key`

Loss of either means encrypted GatewayLink secrets are unrecoverable.

### Two-node Operation

- Each node maintains its own SQLite database
- GatewayLinks must be created on both nodes (upstream on A, downstream on B) with matching secrets
- Rotation must be performed on both nodes
- There is no automatic cross-node synchronization

### Port Requirements

Only ports **80 (TCP)** and **443 (TCP+UDP)** are open on the cloud security group. All cross-server relay traffic goes through port 80 via Caddy.
